# INTAKE：需求澄清

目标：把「角色图 + wiki 1-3 组（必传）± 用户剧本需求（可选）」变成一份用户确认过的 `gdd.md`。

**角色上限 3 组**：超过 3 组时请用户精简——每组角色要产出 6-8 张差分立绘，3 组即 18-24 张生图，再多资产与剧本复杂度都失控。

**按需澄清，至多一轮**：用户没提需求 → 按 wiki 给一轮建议性提问（用户可甩手）；提了需求 → 只问不清楚的缺口。无论哪条路，GDD 确认 gate 本身就是最后一次把关。

## 步骤

### 1. 逐角色提取角色卡（不问用户，先做）

对**每组**角色图+wiki 分别执行：

- 用视觉能力读角色图，输出**外貌锚点描述**（发型/发色、瞳色、服装、气质、标志性配饰）——这是后续所有生图 prompt 的一致性锚点，原话存进 gdd.md 的「角色卡」节（每角色一小节）。
- 从 wiki 提取：姓名、身份、性格、口癖/说话方式、背景故事、关键关系、代表色（若 wiki 无则从角色图取色）。**wiki 未给姓名时**：按人设气质代拟一个（如冷脸舞者=凌霜、沉稳木匠=顾樵），写进角色卡并在建议性提问/缺口确认中告知用户可改——不要卡住等用户起名。
- 多角色时额外判断：**角色间气场关系**（互补/对立/镜像），作为题材建议的依据之一（如冷脸舞者×沉稳木匠 → 双话少慢热，适合治愈/悬疑而非吵闹喜剧）。

### 2. 按用户是否提了剧本需求，分两条路

**路径 A：用户没提剧本需求** → 按 wiki 主动给**一轮建议性提问**（一次问完，带着推断好的答案问，用户零成本可甩手）：题材推荐 2-3 个并说明理由（如"wiki 里胜负欲和别扭的在意，很适合校园恋爱"），关键参数给默认值。用户选择/补充 → 按用户意见出 gdd.md；用户说「不知道 / 你看着办」→ 全部用默认自由发挥，出 gdd.md。

**路径 B：用户提了剧本需求**（题材/梗概/桥段/大纲任意形式）→ 先做**清晰度检查**，只问缺口，一次问完，然后结合角色图+wiki+用户需求出 gdd.md：

- 需求里已明确的（如"悬疑""3 个结局""要有天台戏"）→ 直接采用，不问
- 缺失且影响结构的 → 按下表选项问（题材未定/基调不明/结局数与时长未定/数值设计未定/需求内部矛盾）
- 用户说「你看着办」→ 缺口全部用默认，继续

选项库（路径 A 建议性提问、路径 B 缺口提问，都从这里取）：

| 问题 | 预设选项（第一项为默认） |
|---|---|
| 游戏标题？ | （根据角色名/设定提议一个）/ 用户自定 |
| 玩家视角？ | 第二人称「你」，无名无性别描写 / 自定义主角名 |
| 题材？ | **按 wiki 推断给 2-3 个建议项**：恋爱 / 悬疑推理 / 冒险 / 奇幻 / 科幻 / 日常治愈 / 用户自定 |
| 基调？ | 轻松 / 正剧 / 黑暗致郁（按题材给建议） |
| 角色构成与数值模式？ | 单主角（1 组角色）/ 1 主角+配角·主线模式（变量只追主线）/ 多攻略模式（每角色独立变量，≤3 角色）——按题材建议：恋爱/悬疑可多攻略，治愈/成长多走主线 |
| 结局数量？ | 3 个（好/普通/坏三档）/ 2 个 / 4 个 |
| 单局时长？ | 短（约 10 分钟，4 章）/ 中（约 20 分钟，6 章） |
| 分支数值设计？ | 按题材用通用设计法推导（scriptwriter.md 分支数值系统）：提炼核心张力→选 1-2 个变量→定幅度→反推阈值→检验可达性；现成参考：恋爱=好感度 / 悬疑=信任度+线索数 / 冒险=勇气值 |
| UI 主题？ | **按视觉策略推导法从内容主体推导**（角色代表色×核心场景光源×情绪温度，每参数带推导句，见 build.md 视觉策略推导法）/ 用户指定。禁止"题材→预设主题"查表式匹配，预设只作骨架与兜底 |
| 立绘与 CG 数量？ | 标准（每角色差分×6 + 背景×4 + CG×2）/ 精简（每角色差分×4 + 背景×3）/ 丰富（每角色差分×8 + 背景×6 + CG×4）——多角色时按角色数倍增，3 角色丰富档=24 张立绘，需向用户说明生图耗时 |
| 配音/BGM？ | 仅 BGM / BGM + 关键句配音 / 无 |

### 3. 产出 gdd.md 并请用户确认

结构：

```
# <游戏标题> GDD
## 角色卡（每角色一小节：外貌锚点原文 + wiki 提取设定；多角色附角色间气场关系判断）
## 题材与基调
## 剧情梗概（3-5 句话，含结局走向暗示）
## 用户指定元素（剧情来源非"全创作"时：用户的梗概/必含桥段/大纲原文要点，逐条落实位置）
## 系统规格：视角/角色构成与数值模式（单主角/主线/多攻略）/结局数/章节数/分支数值变量与阈值设计（多攻略=每角色独立变量+组合判定表）
## 结局清单：每结局三要素——结局名（剧情感命名，禁止 HE/BE 代号）/判词（一句定调）/专属 CG 意象（含镜头语言：圆满推近转暖、坏结局拉远降饱和、普通平移低对比）+ 卡面编号 ENDING x/N（用法见 scriptwriter.md「结局达成卡」节）
## 叙事视角与名牌约定：叙述人称（第一「我」/第二「你」/第三人称）+ 主角名牌名（主角说出口的话必须带名牌显示的名字——来自用户输入/wiki；都没有时 agent 起一个贴合题材的名字写在这里，GDD 确认时由用户把关；刻意匿名主角才允许填「无」，此时主角台词也以旁白呈现）+ 旁白定位（客观叙述/主角内心独白）
## 主题配方：骨架主题名 + **逐参数推导**（主色/粒子意象与动效语义/兜底渐变/封面氛围/**HUD 设计**——面板材质/形状语言/名牌形态/正文与旁白排版，每项附一句推导依据——从角色卡与剧情意象出发，写不出依据=没推导，回炉；方法见 build.md 视觉策略推导法、「动效设计」节与「HUD 设计推导」节）
## 资产清单预算：每角色立绘差分/背景/CG/BGM 数量与风格（多角色 ×角色数）
```

**必须等用户明确说「确认/可以/开始」才算过 gate。** 用户提修改则改完再确认。

### 4. 过 gate 后

`state.json` 置 `stage=SCRIPT, gdd_confirmed=true`。
